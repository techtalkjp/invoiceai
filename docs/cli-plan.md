# Git Activity CLI 要件整理（UX/ユースケース中心）

## 目的

月次で請求書を作るタイミングに、ユーザーが「簡単に終わる」と感じる体験を提供する。
目標は「月末に 1 コマンドで締め処理できる」こと。
加えて、Web会員登録からCLI初回同期までを迷わない 1 フローとして成立させること。

## ターゲットユーザー

- 業務委託エンジニア
- クライアントの private repo で日常的に作業
- 複数クライアント、複数repoを横断している可能性が高い

## Web連携を含むオンボーディング方針

- CLI単体ではなく、Web会員登録から連続した体験として設計する
- ユーザーに覚えさせる初回コマンドは `invoiceai setup` のみ
- Webは「請求先の土台を作る」、CLIは「作業repoを紐づけて同期する」役割分担にする

## 初回オンボーディング（最重要）

1. Webで会員登録
2. 組織作成（最小入力）
3. クライアント1件作成（最小入力、後編集前提）
4. Webで `npx invoiceai setup` を大きく案内（コピーボタン）
5. CLI `setup` 実行
6. 未ログインならCLI内で `login` を自動実行
7. repo自動検出とクライアント紐づけ
8. 初回同期実行
9. 完了後、Webで「接続済み」と次アクションを表示

## オンボーディングUX要件

- 迷わせない: ステップ表示（例: `2/5`）を出す
- 入力を減らす: 初回は必須項目を最小化
- 中断耐性: 途中離脱しても再開できる
- 詰まらせない: 未分類repoがあっても完了扱いにする
- 案内を絞る: エラー時は「次の1アクション」だけ表示する

## UX原則

- 指定を減らす: repo や month を毎回指定させない
- 実行回数を減らす: 複数repoでも一括で処理する
- 初期設定を軽くする: 最初だけ対話、以後は自動
- 失敗しにくくする: 冪等で再実行できる
- 処理を止めない: 未分類repoがあっても全体は進める

## 主要ユースケース

1. 初回セットアップ

- `invoiceai setup`
- 未ログインなら setup 内でログインを実行
- クライアントが1件もなければ Web のクライアント作成導線を提示
- repo候補を自動検出し、案件紐づけを保存

2. 日次運用

- `invoiceai sync` で前回以降の差分を同期
- 結果件数だけ確認して終了

3. 複数repo同時運用

- 複数repoの活動を一括同期
- repoごとの保存済み紐づけで自動振り分け

4. 月末締め

- `invoiceai month-close` を実行
- 当月分を案件別に集約して請求作成に進む

5. 紐づけ修正

- repo紐づけを後から更新
- 対象期間を再同期して反映

6. エラー復旧

- 認証切れは再ログイン
- 失敗repoのみ再実行して続行

## 月次の理想フロー（最重要）

1. ユーザーが `invoiceai month-close` を実行
2. CLI が対象repoを自動収集して当月分を同期
3. 案件別の請求候補を表示
4. ユーザーは確認して請求書作成に進む

## コマンド体験要件

- `invoiceai setup`: 初回入口（未ログイン時は認証を含む）
- `invoiceai login`: 認証だけやり直したいときに使用
- `invoiceai sync`: 差分同期（日次）
- `invoiceai month-close`: 月次締め（主役）
- `invoiceai month-close --fix`: 未分類を対話で修正しながら締める
- `invoiceai repo list|add|remove|map`: 後から設定変更

## 初回導線の定義（曖昧さ解消）

- `login` の責務: 認証のみ（トークン取得）
- `setup` の責務: 初回入口（必要なら認証を含む）+ 利用設定（repo検出・repo紐づけ保存）
- 初回利用の標準フロー: `setup` 単独
- `setup` 実行時に未ログインなら:
  - 自動で `login` フローを実行する
  - 認証成功後、そのまま setup を継続する
- `setup` 実行時にクライアントが0件なら:
  - Webのクライアント作成ページURLを表示して作成を促す
  - 作成後に `invoiceai setup` を再実行して続行できるようにする
- `setup` 未実施で `sync` / `month-close` を実行した場合:
  - 自動で `setup` を開始するか確認する
  - 拒否された場合は実行を中断し、`invoiceai setup` を案内する
- `login` は残すが、案内上は補助コマンドとして扱う

## Repo検出要件（ghq非依存）

優先順位:

1. 保存済みrepoリストを使用（メイン）
2. 既定ディレクトリを浅く探索（例: `~/work`, `~/src`, `~/dev`, `~/projects`）
3. カレントディレクトリがgit repoなら対象に含める
4. `ghq` がある場合のみ補助的に利用

補足:

- 毎回フルスキャンしない
- 新規repoのみ差分検出して追加確認

## 初期設定要件

`invoiceai setup` は 3 ステップで終わること:

1. repo候補の提示（デフォルト全選択）
2. repo → クライアント案件の紐づけ
3. 保存して完了メッセージ（次回は `month-close` だけで良いと案内）

ルール:

- 不明repoは「未分類」で先に進める
- セットアップを未分類の存在で失敗させない
- クライアント未登録時は、Web作成導線を出してセットアップを一時停止する

## データと安全性（private repo前提）

送信対象（最小限）:

- timestamp
- repo識別子
- commit hash
- commit title
- author email

送らないもの:

- ソースコード本文
- diff本文

## MVP要件（按分なし前提）

必須:

1. Web会員登録→CLI setup→初回同期の連続導線
2. setup/month-close の 2 主動線（login は補助）
3. 複数repo一括同期
4. repo紐づけ保存と再利用
5. 再実行しても重複しないこと（冪等）
6. 未分類を残したまま月次処理を完了できること

後回し:

1. PR/Review 収集の高度化
2. repo自動分類の賢い推定
3. 高度なフィルタ条件
