# クライアント企業ポータル設計

> **ステータス: 構想段階（未実装）**
>
> このドキュメントは業務委託エンジニアを発注するクライアント企業（ペルソナ D）向けの
> 体験設計です。現時点でこのペルソナ向けの機能はほぼ存在しない。

### 既存実装との関係

|          | 既存 | 本設計                                 |
| -------- | ---- | -------------------------------------- |
| 稼働確認 | なし | 委託先の稼働状況を閲覧（読み取り専用） |
| 請求受領 | なし | 請求書の受領・承認フロー               |
| レポート | なし | 委託先別の月次稼働サマリー             |
| 通知     | なし | 請求書到着通知、稼働レポート定期送信   |

---

## ターゲットユーザー

業務委託エンジニアを複数名抱える企業の担当者（マネージャー/経理）。
委託先がタイムリーに請求書を発行してくれない問題を解決したい。
委託先の稼働状況をリアルタイムに把握し、請求プロセスを効率化したい。

## ゴール

**委託先の稼働状況と請求書を一箇所で確認でき、承認まで完結する。**
委託先に「請求書まだですか？」と催促する必要がなくなること。

## フロー全体像

```
招待受信 → ポータルにログイン → 稼働状況確認 → 請求書受領・承認
```

---

## 導入方法

クライアント企業は委託先（ペルソナ A or B）から招待を受けてポータルにアクセスする。
クライアント企業自身がサインアップするのではなく、委託先が「このクライアントにポータルアクセスを提供する」形。

### 招待フロー

1. 委託先（管理者 or フリーランス）が Settings/Clients でクライアント担当者のメールを登録
2. SendGrid で招待メール送信
3. クライアント担当者がサインアップ → ポータルにアクセス可能に
4. 閲覧権限のみ（クライアントロール）

---

## ポータル機能

### 稼働状況ダッシュボード（読み取り専用）

```
┌─────────────────────────────────────────────────┐
│ 株式会社Example — 2026年2月の稼働状況              │
│                                                   │
│  委託先: coji さん                                 │
│                                                   │
│  稼働日数: 18日                                    │
│  合計稼働: 142.5h                                  │
│  ステータス: 入力中（確定前）                        │
│                                                   │
│  [ 詳細を見る ]                                    │
└─────────────────────────────────────────────────┘
```

- 委託先ごとの月次稼働サマリーを表示
- 稼働データは委託先が入力した内容をそのまま閲覧（編集不可）
- 委託先が「確定」するまでは「入力中」ステータス

### 請求書の受領・承認

```
┌─────────────────────────────────────────────────┐
│ 請求書一覧                                        │
│                                                   │
│  2026年2月分 — coji さん                           │
│    金額: ¥500,000                                  │
│    ステータス: 発行済み                              │
│    [ PDF ] [ 承認する ]                             │
│                                                   │
│  2026年1月分 — coji さん                           │
│    金額: ¥480,000                                  │
│    ステータス: 承認済み → 入金待ち                    │
│    [ PDF ]                                         │
└─────────────────────────────────────────────────┘
```

- 委託先が発行した請求書を一覧表示
- PDF ダウンロード
- 承認ボタンで「承認済み」に変更 → 委託先に通知
- 請求書ステータス: 下書き → 発行済み → 承認済み → 入金済み

### 通知

| イベント                 | 通知方法                   |
| ------------------------ | -------------------------- |
| 請求書が発行された       | メール通知                 |
| 月次稼働レポート（定期） | メール通知（月初に前月分） |
| 稼働が確定された         | メール通知                 |

---

## アクセス権限

| ロール       | できること                                     | できないこと                                 |
| ------------ | ---------------------------------------------- | -------------------------------------------- |
| クライアント | 自分に紐づく委託先の稼働閲覧、請求書閲覧・承認 | 稼働データの編集、他クライアントのデータ閲覧 |

- クライアントは専用のロールを持ち、通常のメンバーとは異なる
- 見られるデータは「自分がクライアントとして登録されている委託先の稼働・請求」のみ

---

## GitHub App 連携（オプション）

クライアント企業が自社 Org に GitHub App をインストールすることで、委託先の GitHub 活動を自動収集できる。

- 企業が自分の Org にインストールするので承認の壁がない
- 委託先メンバーの活動（コミット、PR、レビュー）が自動で可視化される
- 委託先側でタイムシートのドラフトを自動生成 → 確認・修正して確定

詳細は [github-app-migration.md](../github-app-migration.md) を参照。

## 段階的な実装アプローチ

### Phase 1: 最小限のポータル

- クライアント担当者の招待・認証
- 稼働サマリーの読み取り専用ダッシュボード
- 請求書 PDF の閲覧・ダウンロード

### Phase 2: 承認フロー

- 請求書の承認ボタン + ステータス管理
- 承認時の委託先への通知

### Phase 3: 通知・レポート

- 請求書発行時のメール通知
- 月次稼働レポートの定期送信
- 稼働確定時の通知

---

## 前提条件

この機能は以下が先に実装されている必要がある：

- 請求書の freee 非依存化（InvoiceAI 単体での請求書作成・管理）
- 請求書ステータス管理（下書き / 発行済み / 承認済み / 入金済み）
- メール送信基盤（SendGrid）

これらは管理者オンボーディング（[admin.md](admin.md)）の Phase 1, 3 で実装予定。
