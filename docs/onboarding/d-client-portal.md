# D. クライアント企業 — ポータル設計

> **ステータス: 構想段階（未実装）**
>
> このドキュメントは業務委託エンジニアを発注するクライアント企業（ペルソナ D）向けの
> 体験設計です。現時点でこのペルソナ向けの機能はほぼ存在しない。

### 既存実装との関係

|          | 既存 | 本設計                                 |
| -------- | ---- | -------------------------------------- |
| 稼働確認 | なし | 委託先の稼働状況を閲覧（読み取り専用） |
| 請求受領 | なし | 請求書の受領・承認フロー               |
| レポート | なし | 委託先別の月次稼働サマリー             |
| 通知     | なし | 請求書到着通知、稼働レポート定期送信   |

---

## ターゲットユーザー

業務委託エンジニアを複数名抱える企業の担当者（マネージャー/経理）。
委託先がタイムリーに請求書を発行してくれない問題を解決したい。
委託先の稼働状況をリアルタイムに把握し、請求プロセスを効率化したい。

## ゴール

**委託先の稼働状況と請求書を一箇所で確認でき、承認まで完結する。**
委託先に「請求書まだですか？」と催促する必要がなくなること。

## フロー全体像

```
招待受信 → ポータルにログイン → 稼働状況確認 → 請求書受領・承認
```

---

## 導入方法

クライアント企業は委託先（ペルソナ A or B）から招待を受けてポータルにアクセスする。
クライアント企業自身がサインアップするのではなく、委託先が「このクライアントにポータルアクセスを提供する」形。

### 招待フロー

1. 委託先（管理者 or フリーランス）が Settings/Clients でクライアント担当者のメールを入力し「ポータルアクセスを付与」
2. SendGrid で招待メール送信（メンバー招待と同じ SendGrid 基盤を利用）
3. クライアント担当者がリンクをクリック
   - 未登録: サインアップ → 組織が自動作成される → 委託先との紐付けが完了
   - 登録済み: ログイン → 既存の組織ダッシュボードに委託先が追加される
4. クライアント企業のダッシュボードに委託先の稼働・請求が表示される

メンバー招待（better-auth の `invite-member`）とは別の仕組み。クライアントは委託先の組織に参加するのではなく、自分の組織から委託先のデータを閲覧する。紐付けは専用の DB テーブル（例: `client_portal_access`）で管理する。

## ポータル機能

### 稼働状況ダッシュボード（読み取り専用）

**複数の委託先がいる場合:**

```
┌─────────────────────────────────────────────────┐
│ 2026年2月の稼働状況                                │
│                                                   │
│  coji さん           18日  142.5h  ✓ 確定済み     │
│  yamada さん         15日  120.0h  ✓ 確定済み     │
│  tanaka さん          —     —     ⏳ 未確定       │
│                                                   │
│  合計: 33日 / 262.5h（確定分のみ）                 │
└─────────────────────────────────────────────────┘
```

**委託先の詳細ビュー:**

```
┌─────────────────────────────────────────────────┐
│ coji さん — 2026年2月の稼働状況                    │
│                                                   │
│  稼働日数: 18日                                    │
│  合計稼働: 142.5h                                  │
│  ステータス: ✓ 確定済み                             │
│                                                   │
│  [ 日別の詳細を見る ]                              │
└─────────────────────────────────────────────────┘
```

- 委託先ごとの月次稼働サマリーを一覧表示
- 確定済みの稼働のみ数値を表示。未確定の委託先は「未確定」ステータスのみ表示（数値は非表示）
- 稼働データは委託先が入力した内容をそのまま閲覧（編集不可）
- 稼働確定の定義は [b-admin.md](b-admin.md) の「稼働確定と請求フロー」を参照

### 請求書の受領・承認

```
┌─────────────────────────────────────────────────┐
│ 請求書一覧                                        │
│                                                   │
│  2026年2月分 — coji さん                           │
│    金額: ¥500,000                                  │
│    ステータス: 発行済み                              │
│    [ PDF ] [ 承認する ]                             │
│                                                   │
│  2026年1月分 — coji さん                           │
│    金額: ¥480,000                                  │
│    ステータス: 承認済み → 入金待ち                    │
│    [ PDF ]                                         │
└─────────────────────────────────────────────────┘
```

- 委託先が発行した請求書を一覧表示
- PDF ダウンロード
- 承認ボタンで「承認済み」に変更 → 委託先に通知
- 請求書ステータスの定義は [b-admin.md](b-admin.md) の「請求書ステータス」を参照

### 通知

| イベント                 | 通知方法                   |
| ------------------------ | -------------------------- |
| 請求書が発行された       | メール通知                 |
| 月次稼働レポート（定期） | メール通知（月初に前月分） |
| 稼働が確定された         | メール通知                 |

## アクセス権限

### ロール設計

better-auth の organization plugin が持つ標準ロールは `owner` / `admin` / `member` の 3 つ。
クライアント企業向けには以下の 2 つのアプローチが考えられる：

| 方式               | 概要                                                                               | メリット                       | デメリット                       |
| ------------------ | ---------------------------------------------------------------------------------- | ------------------------------ | -------------------------------- |
| 別組織方式         | クライアント企業を独立した組織として作成し、委託先の稼働データを共有リンク等で閲覧 | better-auth のロールを変更不要 | 組織間のデータ共有の仕組みが必要 |
| カスタムロール方式 | better-auth のロールに `client` を追加し、同一組織内でアクセス制御                 | データ共有が不要               | better-auth のロール拡張が必要   |

**方針: 別組織方式を採用する。** 理由：

- クライアント企業は委託先の組織に所属するのではなく、自社の文脈でデータを閲覧する
- 複数の委託先を抱えるクライアント企業は、委託先ごとに組織に参加するのは非現実的
- better-auth の標準ロールを壊さずに済む

### 別組織方式の仕組み

1. クライアント企業は独自の組織を持つ（サインアップ時に自動作成）
2. 委託先がクライアントに「ポータルアクセス」を付与すると、紐付けが作られる
3. クライアント企業の組織ダッシュボードに、紐付けられた委託先の稼働・請求が表示される
4. 閲覧のみ（委託先の組織のデータを編集する権限はない）

| ロール      | 組織                   | できること                                                 |
| ----------- | ---------------------- | ---------------------------------------------------------- |
| owner/admin | 委託先の組織           | 稼働入力、請求書作成、クライアントへのポータルアクセス付与 |
| member      | 委託先の組織           | 稼働入力                                                   |
| owner/admin | クライアント企業の組織 | 紐付けられた委託先の稼働閲覧、請求書閲覧・承認             |

## GitHub 活動の可視化

稼働データの収集は委託先側の責務。クライアントポータルには収集済みのデータが反映される。

- **クライアント Org の private repo で作業している場合** — 委託先メンバーが CLI（`invoiceai sync`）でローカルから収集
- **委託先の自社 Org の private repo で作業している場合** — 委託先が GitHub App をインストールしてサーバーサイドで自動収集

クライアント企業側は収集方式に関わらず、確定済みの稼働データを閲覧するのみ（詳細は [a-freelance.md](a-freelance.md) を参照）。

## 段階的な実装アプローチ

### Phase 1: 最小限のポータル

- クライアント担当者の招待・認証
- 稼働サマリーの読み取り専用ダッシュボード
- 請求書 PDF の閲覧・ダウンロード

### Phase 2: 承認フロー

- 請求書の承認ボタン + ステータス管理
- 承認時の委託先への通知

### Phase 3: 通知・レポート

- 請求書発行時のメール通知
- 月次稼働レポートの定期送信
- 稼働確定時の通知

---

## 前提条件

この機能は以下が先に実装されている必要がある：

- 請求書の freee 非依存化（InvoiceAI 単体での請求書作成・管理）
- 請求書ステータス管理（下書き / 発行済み / 承認済み / 入金済み）
- メール送信基盤（SendGrid）

これらは管理者オンボーディング（[b-admin.md](b-admin.md)）の Phase 1, 3 で実装予定。
