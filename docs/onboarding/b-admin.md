# B. 管理者 — オンボーディングフロー設計

> **ステータス: 設計中（未実装）**
>
> このドキュメントは小規模受託開発会社の管理者（ペルソナ B）向けの
> オンボーディング体験の設計仕様です。

### 既存実装との関係

|                | 既存（setup フロー）                           | 本設計（管理者オンボーディング）                      |
| -------------- | ---------------------------------------------- | ----------------------------------------------------- |
| ステップ数     | 3ステップ（会社名 → クライアント → CLI）       | 3ステップ（会社名 → クライアント → 完了）             |
| freee 連携     | 設定画面で手動設定                             | 同様（オプション、会計連携として位置づけ）            |
| メンバー招待   | 登録済みユーザーのみ直接追加                   | better-auth の invitation API + SendGrid でメール招待 |
| GitHub 連携    | 設定画面で手動設定                             | メンバーに CLI セットアップを案内（下記参照）         |
| セットアップ後 | ダッシュボードにリダイレクト（ガイダンスなし） | TODO チェックリストで次のアクションを案内             |

> **GitHub 連携について**: リポジトリの所在によって収集方式が異なる。
>
> - **クライアント Org の private repo** — 管理者が自社 Org に GitHub App をインストールしても連携できない。各メンバーがローカル環境で CLI（`invoiceai sync`）を使い `git log` / `gh` CLI 経由で収集する。
> - **自社 Org の private repo** — GitHub App を自社 Org にインストールすればサーバーサイドで自動収集できる。
>
> CLI と GitHub App は併用可能。どちらで収集されたアクティビティも同じ `activity` テーブルに統合される（詳細は [a-freelance.md](a-freelance.md) を参照）。

### 既存から流用できる部分

- 認証フロー（メール/パスワード サインアップ）
- 組織自動作成（`ensureUserHasOrganization()`）
- クライアント作成フォーム（`client-form.tsx`）
- better-auth の `invitation` テーブル + API（既にスキーマ定義済み）
- CLI セットアップフロー（メンバー向け、ペルソナ A の [a-freelance.md](a-freelance.md) を参照）

---

## ターゲットユーザー

3〜10 名程度の受託開発会社の管理者（社長/マネージャー）。
メンバーの稼働管理とクライアントへの月次請求を効率化したい。

## ゴール

**サインアップから「自分1人で稼働管理を始められる」まで 3 分。**
まず1人で使い始め、慣れてからメンバーを招待する自然な流れを提供する。
セットアップ完了後に「あとは何をすべきか」が明確にわかること。

## フロー全体像

```
サインアップ → 会社名確認 → クライアント登録 → 完了
```

この 3 ステップで「使える状態」にする。メンバー招待は完了後の TODO で案内。

---

## 詳細フロー

### Step 1: サインアップ + 会社名確認

```
InvoiceAI へようこそ！

会社名 / 屋号を確認してください:
  [ coji の組織           ]

  ✓ 次へ
```

- サインアップ完了時に組織を自動作成（既存通り）
- デフォルト名「{ユーザー名} の組織」→ 正式名に変更を促す
- **既存実装あり**: `setup/company.tsx`

### Step 2: クライアント登録

```
最初のクライアント（請求先）を登録しましょう。

  クライアント名: [ 株式会社ABC          ]
  請求タイプ:     ( ) タイムチャージ  (●) 固定月額
  月額:           [ 500,000             ] 円

  ✓ 登録
```

- **既存実装あり**: `setup/client.tsx`
- 請求タイプ・単価の設定を初回に案内（後から変更可能）
- freee 取引先 ID はここでは不要（freee 連携は別途オプション）

### Step 3: 完了 + ガイダンス

```
🎉 セットアップ完了！

まずは自分の稼働を入力してみましょう。
慣れたらメンバーを招待して、チームで使い始められます。

次にやること:

  □ 今月の稼働時間を入力する
  □ メンバーを招待する
  □ 請求書を作成する

  [ ダッシュボードへ ]
```

- 完了画面に TODO チェックリストを表示（必須項目のみ）
- まず自分1人で稼働入力を体験させ、価値を実感してからメンバー招待へ誘導
- CLI セットアップ・freee 連携はダッシュボード TODO（任意枠）で案内。完了画面には含めない（情報量を絞る）
- ダッシュボードにも同じ TODO を表示（全部完了するまで）

## セットアップ後のダッシュボード

### 管理者向け TODO バナー

```
┌─────────────────────────────────────────────────┐
│ 💡 はじめに                                       │
│                                                   │
│  ✓ 会社情報を設定しました                          │
│  ✓ クライアントを登録しました                      │
│  □ 今月の稼働時間を入力する →                      │
│  □ メンバーを招待する →                            │
│  □ 最初の請求書を作成する →                        │
│                                                   │
│  💡 もっと便利に:                                   │
│  □ CLI をセットアップする →                        │
│  □ freee と連携する →                              │
│                                                   │
│  [ このバナーを閉じる ]                             │
└─────────────────────────────────────────────────┘
```

- 各 TODO はリンク付きで、対応する画面に遷移
- 必須 TODO と任意 TODO を視覚的に分離（「もっと便利に:」セクション）
- 「CLI をセットアップする」→ 管理者自身がローカルの作業リポジトリで `npx invoiceai` を実行する手順への導線
- 任意 TODO は対応機能が未実装の場合は非表示にする（Phase ごとに段階的に表示項目を増やす）
- 全必須 TODO 完了 or 明示的に閉じるまで表示
- 進捗は `organization.metadata` に保存（既存の `setupCompletedUserIds` パターンを踏襲）
  - `onboarding.completedSteps` に完了済みステップ名を配列で保持
  - `onboarding.dismissedAt` でバナーの非表示状態を管理

### TODO コンプリート判定

| TODO                   | 完了条件                                              |
| ---------------------- | ----------------------------------------------------- |
| 稼働時間を入力する     | 当月に1件以上の稼働エントリが存在する                 |
| メンバーを招待する     | invitation テーブルに1件以上の招待が存在する          |
| 請求書を作成する       | invoice テーブルに1件以上の請求書が存在する           |
| CLI をセットアップする | activity テーブルに CLI 経由のデータが1件以上存在する |
| freee と連携する       | freee の認証トークンが設定済み                        |

## メンバー招待の詳細設計

### better-auth の既存機能を活用

DB テーブル `invitation` は `db/schema.sql` に既にスキーマ定義済み。organization_id, email, role, status, inviter_id, expires_at 等のカラムを持つ。

### 使用する better-auth API

| 操作               | エンドポイント                            | 用途                           |
| ------------------ | ----------------------------------------- | ------------------------------ |
| 招待作成           | `POST /organization/invite-member`        | メンバーを招待                 |
| 招待承認           | `POST /organization/accept-invitation`    | メンバーが招待を承認           |
| 招待拒否           | `POST /organization/reject-invitation`    | メンバーが招待を拒否           |
| 招待キャンセル     | `POST /organization/cancel-invitation`    | 管理者が招待を取り消し         |
| 招待一覧           | `GET /organization/list-invitations`      | 管理者が招待状況を確認         |
| ユーザーの招待一覧 | `GET /organization/list-user-invitations` | メンバーが受け取った招待を確認 |
| 招待詳細           | `GET /organization/get-invitation`        | 招待リンクからの情報取得       |

### メール送信（SendGrid）

- better-auth の `sendInvitationEmail` コールバック（`app/lib/auth.ts`）で SendGrid を使用
- 招待者名・組織名を含む件名と、承認リンク付きの HTML メールを送信
- 環境変数 `SENDGRID_API_KEY` が必要

## 稼働確定と請求フロー

### 稼働ステータス

稼働データには「確定」の概念を導入する。

| ステータス | 意味                                       | 操作者             |
| ---------- | ------------------------------------------ | ------------------ |
| 入力中     | スタッフが稼働時間を入力・修正できる状態   | スタッフ           |
| 確定済み   | その月の稼働を確定。以降は管理者のみ修正可 | スタッフ or 管理者 |

- スタッフは月末に「確定」ボタンで自分の稼働を確定する
- 管理者は全スタッフの確定状況を一覧で確認できる
- 確定済みの稼働からタイムシート PDF を生成し、請求書に紐づける
- クライアントポータル（ペルソナ D）では確定済みの稼働のみ閲覧可能

### 請求書ステータス

| ステータス | 意味                                                     |
| ---------- | -------------------------------------------------------- |
| 下書き     | 管理者が作成中。まだ発行していない                       |
| 発行済み   | クライアントに送付済み                                   |
| 承認済み   | クライアントが内容を承認した（クライアントポータル経由） |
| 入金済み   | 入金を確認した                                           |

- 「承認済み」はクライアントポータル（ペルソナ D）があるときのみ使用
- クライアントポータルがない場合は「発行済み → 入金済み」の 3 段階で運用可能

### 請求書機能の位置づけ

現在の請求書作成は freee 連携が前提になっている。freee がなくても請求書を発行・管理できるようにする。

| 機能       | freee なし                       | freee あり               |
| ---------- | -------------------------------- | ------------------------ |
| 請求書作成 | InvoiceAI 単体で作成・PDF 出力   | 同左                     |
| 請求書管理 | InvoiceAI 内でステータス管理     | 同左                     |
| 会計連携   | なし                             | freee に請求データを同期 |
| 取引先管理 | InvoiceAI のクライアント機能のみ | freee 取引先とマッピング |

- freee 連携は「会計連携」として位置づけ、オプション扱い
- 請求書の作成・管理は InvoiceAI 単体で完結させる
- freee 連携時は既存の同期機能を使って会計データを送信

## 実装の優先順位

### Phase 1: オンボーディング体験（最優先）

1. **セットアップ Step 3** — `setup/complete.tsx`（完了 + TODO ガイダンス）
2. **ダッシュボード TODO バナー** — 管理者向け / スタッフ向け
3. **セットアップナビゲーション更新** — 3 ステップ対応

### Phase 2: 招待機能

1. **SendGrid 基盤** — `sendInvitationEmail` コールバック + メールテンプレート
2. **招待承認ルート** — `/invite/:invitationId`（未登録/登録済みの分岐）
3. **Settings/Members の改修** — 直接追加 → 招待ベースに移行、招待一覧表示

### Phase 3: 請求書独立化

1. **請求書の freee 依存を除去** — freee なしでも作成・PDF 出力可能に
2. **請求書ステータス管理** — 下書き / 発行済み / 入金済み
3. **freee 連携をオプション化** — 「会計連携」として Settings に配置
