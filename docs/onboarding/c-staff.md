# C. スタッフ — オンボーディングフロー設計

> **ステータス: 設計中（未実装）**
>
> このドキュメントは受託開発会社のスタッフ / メンバー（ペルソナ C）向けの
> オンボーディング体験の設計仕様です。
> 管理者による招待フローは [b-admin.md](b-admin.md) を参照。

### 既存実装との関係

|          | 既存                                             | 本設計                                       |
| -------- | ------------------------------------------------ | -------------------------------------------- |
| 参加方法 | 管理者が Settings/Members から直接追加           | メール招待 → 承認で参加                      |
| 初回体験 | ログイン後すぐにダッシュボード（ガイダンスなし） | 参加画面 → TODO バナーで次のアクションを案内 |
| 稼働入力 | 手動のみ                                         | 手動 + GitHub 活動からの自動提案             |

---

## ターゲットユーザー

受託開発会社に所属するエンジニア・デザイナー。
管理者から招待メールを受け取り、日々の稼働時間を入力する。
できれば GitHub の活動から自動入力されて、確認だけで済むのが理想。

## ゴール

**招待メールから稼働入力開始まで 2 分。**
「何をすればいいか」が明確で、迷わない体験にする。

## フロー全体像

```
招待メール受信 → 承認 → 稼働入力開始
```

---

## 詳細フロー

### Step 1: 招待メール受信

管理者が招待を送信すると、SendGrid 経由でメールが届く。
招待の仕組み（better-auth API、SendGrid 設定）の詳細は [b-admin.md](b-admin.md) の「メンバー招待の詳細設計」を参照。

```
件名: tanaka さんから 株式会社Example への招待

──────────────────────────────
InvoiceAI

tanaka さんが あなたを 株式会社Example に招待しました。

InvoiceAI は稼働時間の管理と請求書作成のサービスです。
招待を承認すると、稼働時間の入力を開始できます。

  [ 招待を承認する ]

※ このリンクの有効期限は 48 時間です。
──────────────────────────────
```

### Step 2: 招待承認

メールのリンク `/invite/:invitationId` をクリックした後の分岐：

```
/invite/:invitationId
  ↓
├─ 未登録ユーザー
│    招待内容を表示（組織名、招待者名）
│    → [ サインアップして参加 ]
│    → /auth/signup?callbackURL=/invite/:invitationId
│    → サインアップ完了後、自動で招待承認 → 組織に参加
│    → /org/:orgSlug にリダイレクト
│
└─ 登録済みユーザー
     ├─ ログイン済み
     │    招待内容を表示
     │    → [ 参加する ] / [ 辞退する ]
     │    → 承認: POST /organization/accept-invitation
     │    → /org/:orgSlug にリダイレクト
     │
     └─ 未ログイン
          → /auth/signin?callbackURL=/invite/:invitationId
          → ログイン後、招待内容を表示 → 承認
```

- better-auth の `GET /organization/get-invitation` で招待詳細を取得して表示
- 未登録の場合はサインアップ後に自動承認（callbackURL で招待ページに戻る）
- 期限切れの場合はエラー表示 + 管理者への再招待依頼を案内

### Step 3: 参加完了 + ガイダンス

```
🎉 株式会社Example に参加しました！

  あなたのロール: メンバー

  まず稼働時間を入力しましょう:
  → 今月の稼働入力 (/org/example/work-hours)

  💡 作業リポジトリで npx invoiceai を実行すると、
     Git の活動から稼働時間を自動で取得できます。
```

- 参加直後に稼働入力画面への導線を出す
- CLI セットアップは任意（手動入力のみでも利用可能）
- CLI の案内は `npx invoiceai` の実行方法を直接示す（別ページへの遷移は不要）

## ダッシュボード TODO バナー

```
┌─────────────────────────────────────────────────┐
│ 💡 はじめに                                       │
│                                                   │
│  ✓ 株式会社Example に参加しました                  │
│  □ 今月の稼働時間を入力する →                      │
│  □ タイムシートを確認する →                        │
│                                                   │
│  💡 もっと便利に:                                   │
│  □ CLI をセットアップする →                        │
│                                                   │
│  [ このバナーを閉じる ]                             │
└─────────────────────────────────────────────────┘
```

- 管理者向けより項目が少ない（簡潔に）
- 必須 TODO と任意 TODO を視覚的に分離（「もっと便利に:」セクション、管理者バナーと同じパターン）
- 「CLI をセットアップする」→ `npx invoiceai` の手順を案内するインラインガイドを表示
- 進捗は `organization.metadata` の `onboarding.users.{userId}.completedSteps` で管理（管理者バナーと同じ `organization.metadata` に統一）
- 全必須 TODO 完了 or 明示的に閉じるまで表示

## 日常の稼働入力フロー

### 手動入力

Web UI (`/org/:orgSlug/work-hours`) で日別に開始時刻・終了時刻・休憩・作業内容を入力。
既存実装をそのまま利用。

### GitHub 活動からの自動提案

リポジトリの所在によって収集方式が異なる：

- **クライアント Org の private repo** — CLI（`invoiceai sync`）でローカルの `git log` + `gh` CLI から収集
- **自社 Org の private repo** — GitHub App がインストール済みならサーバーサイドで自動収集

どちらの方式で収集されたアクティビティも同じ流れで処理される：

1. アクティビティがサーバーに蓄積される（CLI 送信 or GitHub App 自動収集）
2. ルールベース + AI（Gemini Flash）で稼働時間を推定
3. スタッフは稼働入力画面で提案を確認 → 承認 or 修正して保存

**稼働入力画面での提案 UI:**

```
┌─────────────────────────────────────────────────┐
│ 2月15日（水）                                     │
│                                                   │
│  💡 GitHub の活動から稼働を推定しました:            │
│    開始: 10:00  終了: 19:00  休憩: 1h              │
│    作業内容: ログイン機能のバグ修正、PR レビュー    │
│                                                   │
│    [ この内容で入力 ]  [ 修正して入力 ]             │
│                                                   │
└─────────────────────────────────────────────────┘
```

- 未入力の日にアクティビティがある場合、提案バナーを表示
- 「この内容で入力」→ 推定値をそのまま稼働エントリとして保存
- 「修正して入力」→ 推定値をフォームに事前入力した状態で編集画面を開く
- 既に手動入力済みの日には提案を表示しない

推定ルールの詳細は [a-freelance.md](a-freelance.md) の「稼働時間の推定ロジック」を参照。

### 月末の稼働確定

月末にスタッフが「確定」ボタンで当月の稼働を確定する。確定後は管理者のみ修正可能。
稼働確定の詳細は [b-admin.md](b-admin.md) の「稼働確定と請求フロー」を参照。

## エッジケース

| ケース                          | 対応                                                        |
| ------------------------------- | ----------------------------------------------------------- |
| 招待リンクの期限切れ（48 時間） | エラー表示 + 「管理者に再招待を依頼してください」を案内     |
| 既に別の組織に所属              | 複数組織への所属を許可（既存仕様）                          |
| 招待メールが届かない            | Settings/Members で招待を再送可能（管理者操作）             |
| 招待を辞退                      | `POST /organization/reject-invitation` で辞退。管理者に通知 |

## 実装の優先順位

1. **招待承認ルート** — `/invite/:invitationId`（未登録/登録済みの分岐）
2. **参加完了画面** — ガイダンス表示 + 稼働入力への導線
3. **ダッシュボード TODO バナー** — スタッフ向け
