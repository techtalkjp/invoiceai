# InvoiceAI プロダクト戦略

## ペルソナ

### A. フリーランス / 個人事業主 / 副業エンジニア（1人）

自分の稼働時間を記録し、クライアントへの請求書を作成・送付したい。
GitHub の活動ログから稼働実績を自動で集計し、手入力の手間を減らしたい。
副業の場合は本業の合間に限られた時間で作業するため、稼働記録の自動化ニーズが特に強い。

### B. 小規模受託開発会社の管理者

複数メンバーの稼働をクライアント別に管理し、請求書を一括生成したい。
メンバーの稼働状況を把握し、未入力のフォローや月次レポートを確認したい。

### C. 受託開発会社のスタッフ（メンバー）

日々の稼働時間を手早く入力し、タイムシートを確認したい。
できれば GitHub の活動から自動で入力されて、確認だけで済むのが理想。

### D. 業務委託エンジニアを複数抱えるクライアント企業

業務委託スタッフがタイムリーに請求書を発行してくれない問題を解決したい。
委託先の稼働状況をリアルタイムに把握し、請求プロセスを効率化したい。

## ペルソナ別 機能マトリクス

### A. フリーランス / 個人事業主 / 副業エンジニア

| 分類             | 既存機能                            | 不足機能                                        |
| ---------------- | ----------------------------------- | ----------------------------------------------- |
| 稼働管理         | 日別の稼働時間入力、タイムシートPDF | -                                               |
| 請求             | 請求書作成、freee連携、PDF出力      | -                                               |
| GitHub           | リポジトリ紐付け、活動から稼働提案  | CLI + GitHub App 併用（自社/クライアント repo） |
| CLI              | login, setup, whoami                | `invoiceai` 1コマンド統合、`invoiceai sync`     |
| オンボーディング | setup フロー                        | -                                               |

概ね揃っている。CLI の `invoiceai` 1コマンドで状態判定 → 同期まで完結させる。

### B. 小規模受託開発会社の管理者

| 分類         | 既存機能                      | 不足機能                                     |
| ------------ | ----------------------------- | -------------------------------------------- |
| メンバー管理 | 直接追加、ロール管理          | メール招待（better-auth invitation API）     |
| 稼働管理     | 全スタッフの月別稼働一覧      | メンバー別の稼働状況サマリー、未入力アラート |
| 請求         | クライアント別請求書、一括PDF | 月次一括請求書生成、請求ステータス管理       |
| GitHub       | GitHub App + CLI 併用         | メンバーの収集状況の管理画面                 |
| クライアント | CRUD、freeeインポート         | -                                            |
| レポート     | なし                          | 月次売上サマリー、クライアント別収益レポート |

管理ダッシュボード系が薄い。「今月の全体像をパッと見たい」ニーズに応えられていない。

### C. 受託開発会社のスタッフ（メンバー）

| 分類     | 既存機能                  | 不足機能                               |
| -------- | ------------------------- | -------------------------------------- |
| 稼働入力 | Web UI で日別入力         | モバイル対応、入力リマインダー         |
| 確認     | タイムシートPDF           | 自分の月次サマリー（合計時間、残業等） |
| GitHub   | 活動から稼働提案          | 自動入力（提案の承認だけで済む）       |
| CLI      | setup（リポジトリ紐付け） | `invoiceai sync`（稼働データ同期）     |

入力の手間を減らす方向の機能が不足。「GitHub活動 → 稼働提案 → 確認だけ」が理想。

### D. 業務委託を発注するクライアント企業

| 分類     | 既存機能 | 不足機能                                           |
| -------- | -------- | -------------------------------------------------- |
| 稼働確認 | なし     | 委託先の稼働状況閲覧（読み取り専用ダッシュボード） |
| 請求受領 | なし     | 請求書の受領・承認フロー                           |
| レポート | なし     | 委託先別の月次稼働サマリー                         |
| 通知     | なし     | 請求書到着通知、稼働レポート定期送信               |

**現状ほぼ機能がない。** ここが一番のギャップ。

## 実装ロードマップ

### Phase 1: 基盤整備（A + B の共通基盤）

| 項目                    | 対象ペルソナ | 概要                                 |
| ----------------------- | ------------ | ------------------------------------ |
| SendGrid メール基盤     | B, C         | 招待メール送信の基盤                 |
| 請求書の freee 非依存化 | A, B         | InvoiceAI 単体で請求書作成・PDF 出力 |
| 請求書ステータス管理    | A, B         | 下書き / 発行済み / 入金済み         |

### Phase 2: メンバー招待 + スタッフ体験（B + C を並行）

| 項目                                                       | 対象ペルソナ | 概要                                   |
| ---------------------------------------------------------- | ------------ | -------------------------------------- |
| 招待機能（better-auth invitation API）                     | B, C         | メール招待 → 承認で参加                |
| 招待承認ルート                                             | C            | `/invite/:invitationId`                |
| Settings/Members の改修                                    | B            | 直接追加 → 招待ベースに移行            |
| セットアップ完了 + TODO ガイダンス                         | B            | Step 3: 完了画面 + ダッシュボード TODO |
| ダッシュボード TODO バナー                                 | B, C         | 管理者向け / スタッフ向け              |
| 招待 UI（Settings/Members + ダッシュボード TODO から誘導） | B            | メンバー招待は初回セットアップ後に案内 |

### Phase 3: CLI オンボーディング（A を並行）

| 項目                              | 対象ペルソナ | 概要                                  |
| --------------------------------- | ------------ | ------------------------------------- |
| `invoiceai`（引数なし）統合フロー | A            | ログイン + リポジトリ設定 + 初回同期  |
| `invoiceai sync` 差分同期         | A            | コミット + gh CLI 経由の PR/レビュー  |
| `POST /api/cli/sync` サーバー API | A            | アクティビティ受信 + 稼働サマリー応答 |

### Phase 4: クライアントポータル（D）

| 項目                        | 対象ペルソナ | 概要                     |
| --------------------------- | ------------ | ------------------------ |
| 稼働確定フロー              | B, C         | スタッフの月次稼働確定   |
| クライアント招待 + ポータル | D            | 稼働閲覧 + 請求書受領    |
| 請求書承認フロー            | D            | 承認ステータスの追加     |
| 通知・レポート              | D            | メール通知、月次レポート |

### 判断基準

- Phase 1 は全ペルソナの前提条件なので最優先
- Phase 2 と 3 は独立しているため並行可能。ただし B + C は SendGrid 基盤に依存するため Phase 1 の後
- Phase 4 は Phase 1〜3 の成果物（請求書管理、稼働確定、メール基盤）が前提
