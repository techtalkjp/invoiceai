# InvoiceAI プロダクト戦略

## 戦略方針

### 原則: 自分が使い倒してから広げる

InvoiceAI は **ペルソナ A（自分＝フリーランス）の実運用定着を最優先** とする。

1. **A を完璧にする** — 自分の月次請求フローが CLI 同期 → 請求書発行まで完全に回ること
2. **A が安定したら B/C** — チーム利用に拡大（招待・メンバー管理）
3. **A〜C が定着したら D** — クライアント企業向けポータル

この順序を守る理由:

- 自分が毎月使う道具として磨かなければ、他人に勧められない
- A の運用品質が B/C の信頼性の基盤になる（同じ稼働管理・請求基盤を使う）
- D は A〜C の成果物（請求書管理、稼働確定、メール基盤）が前提であり、先行しても動かない

### ペルソナ A の完成定義（DoD）

「完成」とは以下がすべて満たされた状態:

1. **月次フローが自動化されている**: 作業リポジトリで `invoiceai sync` → Web で稼働確認 → 請求書 PDF 生成まで、手動介入が最小限
2. **CLI 同期が信頼できる**: 同期成功率 95% 以上、手修正率（同期後に Web で稼働を手動修正する割合）20% 以下
3. **月次処理時間が短い**: 月末の請求書発行まで 15 分以内で完了（以前の手作業比 1/3 以下）
4. **3ヶ月連続で実運用**: 上記を 3ヶ月連続で達成し、運用が安定していること

## ペルソナ

### A. フリーランス / 個人事業主 / 副業エンジニア（1人）

自分の稼働時間を記録し、クライアントへの請求書を作成・送付したい。
GitHub の活動ログから稼働実績を自動で集計し、手入力の手間を減らしたい。
副業の場合は本業の合間に限られた時間で作業するため、稼働記録の自動化ニーズが特に強い。

### B. 小規模受託開発会社の管理者

複数メンバーの稼働をクライアント別に管理し、請求書を一括生成したい。
メンバーの稼働状況を把握し、未入力のフォローや月次レポートを確認したい。

### C. 受託開発会社のスタッフ（メンバー）

日々の稼働時間を手早く入力し、タイムシートを確認したい。
できれば GitHub の活動から自動で入力されて、確認だけで済むのが理想。

### D. 業務委託エンジニアを複数抱えるクライアント企業

業務委託スタッフがタイムリーに請求書を発行してくれない問題を解決したい。
委託先の稼働状況をリアルタイムに把握し、請求プロセスを効率化したい。

## ペルソナ別 機能マトリクス

### A. フリーランス / 個人事業主 / 副業エンジニア

| 分類             | 既存機能                            | 不足機能                                        | 優先度 | Impact/Effort |
| ---------------- | ----------------------------------- | ----------------------------------------------- | ------ | ------------- |
| 稼働管理         | 日別の稼働時間入力、タイムシートPDF | -                                               | -      | -             |
| 請求             | 請求書作成、freee連携、PDF出力      | -                                               | -      | -             |
| GitHub           | リポジトリ紐付け、活動から稼働提案  | CLI + GitHub App 併用（自社/クライアント repo） | P0     | High / Med    |
| CLI              | login, setup, whoami                | `invoiceai` 1コマンド統合、`invoiceai sync`     | P0     | High / Med    |
| オンボーディング | setup フロー                        | -                                               | -      | -             |

**現状評価**: 稼働管理・請求の基本機能は動作する。ボトルネックは CLI 同期。現在の CLI は GitHub App Installation Token（サーバーサイド）前提で、クライアント Org の private repo に対応できない。ローカル `git log` + `gh` CLI ベースに書き直すことで、あらゆるリポジトリの稼働を収集可能になる。

### B. 小規模受託開発会社の管理者

| 分類         | 既存機能                      | 不足機能                                     | 優先度 | Impact/Effort |
| ------------ | ----------------------------- | -------------------------------------------- | ------ | ------------- |
| メンバー管理 | 直接追加、ロール管理          | メール招待（better-auth invitation API）     | P1     | High / Low    |
| 稼働管理     | 全スタッフの月別稼働一覧      | メンバー別の稼働状況サマリー、未入力アラート | P2     | Med / Med     |
| 請求         | クライアント別請求書、一括PDF | 月次一括請求書生成、請求ステータス管理       | P1     | High / Med    |
| GitHub       | GitHub App + CLI 併用         | メンバーの収集状況の管理画面                 | P2     | Low / Med     |
| クライアント | CRUD、freeeインポート         | -                                            | -      | -             |
| レポート     | なし                          | 月次売上サマリー、クライアント別収益レポート | P3     | Med / High    |

**現状評価**: メンバーの直接追加とクライアント管理は動作する。不足の中心は「招待フロー」と「請求ステータス管理」。管理ダッシュボード（サマリー・レポート）は運用上のペインだが、A の定着後に着手すべき（現時点で利用者が自分1人のため）。

### C. 受託開発会社のスタッフ（メンバー）

| 分類     | 既存機能                  | 不足機能                               | 優先度 | Impact/Effort |
| -------- | ------------------------- | -------------------------------------- | ------ | ------------- |
| 稼働入力 | Web UI で日別入力         | モバイル対応、入力リマインダー         | P2     | Med / High    |
| 確認     | タイムシートPDF           | 自分の月次サマリー（合計時間、残業等） | P2     | Med / Low     |
| GitHub   | 活動から稼働提案          | 自動入力（提案の承認だけで済む）       | P1     | High / Med    |
| CLI      | setup（リポジトリ紐付け） | `invoiceai sync`（稼働データ同期）     | P1     | High / Med    |

**現状評価**: 手動入力は動作するが「入力の手間」がボトルネック。GitHub 活動からの自動提案（承認だけで済む）が最大の価値だが、これは A の CLI 同期基盤に依存する。A が安定してから C 向けに展開する。

### D. 業務委託を発注するクライアント企業

| 分類     | 既存機能 | 不足機能                                           | 優先度 | Impact/Effort |
| -------- | -------- | -------------------------------------------------- | ------ | ------------- |
| 稼働確認 | なし     | 委託先の稼働状況閲覧（読み取り専用ダッシュボード） | P3     | High / High   |
| 請求受領 | なし     | 請求書の受領・承認フロー                           | P3     | High / High   |
| レポート | なし     | 委託先別の月次稼働サマリー                         | P3     | Med / Med     |
| 通知     | なし     | 請求書到着通知、稼働レポート定期送信               | P3     | Med / Med     |

**現状評価**: 機能ゼロ。ギャップは最大だが、新規性・実装コストも最大。別組織方式のアクセス制御、組織間データ共有など、アーキテクチャ変更を伴う。A〜C の稼働管理・請求基盤が安定してからでないと、ポータルに表示するデータ自体の品質が保証できない。

## 実装ロードマップ

### Phase 1: A を動かす（CLI 書き直し + 請求基盤）

**目的**: 自分（ペルソナ A）の月次フローを CLI 同期 → 請求書発行まで完結させる。

| 項目                              | 対象ペルソナ | 概要                                  |
| --------------------------------- | ------------ | ------------------------------------- |
| CLI 書き直し（`invoiceai` 統合）  | A            | ログイン + リポジトリ設定 + 初回同期  |
| `invoiceai sync` 差分同期         | A            | ローカル git log + gh CLI → サーバー  |
| `POST /api/cli/sync` サーバー API | A            | アクティビティ受信 + 稼働サマリー応答 |
| 請求書の freee 非依存化           | A, B         | InvoiceAI 単体で請求書作成・PDF 出力  |
| 請求書ステータス管理              | A, B         | 下書き / 発行済み / 入金済み          |

**成功指標（KPI）**:

| 指標         | 目標値 | 計測方法                                               |
| ------------ | ------ | ------------------------------------------------------ |
| 同期成功率   | ≥ 95%  | sync コマンドがエラーなく完了した回数 / 実行回数       |
| 手修正率     | ≤ 20%  | sync 後に Web で稼働を手動修正した日数 / sync 対象日数 |
| 月次処理時間 | ≤ 15分 | 月末の sync 〜 請求書 PDF 生成完了までの所要時間       |
| セットアップ | ≤ 3分  | 初回 invoiceai 実行 〜 初回 sync 完了までの所要時間    |

**次 Phase へのゲート条件**: 自分が 1ヶ月間 Phase 1 の機能で月次請求を完了できること。

### Phase 2: B/C を動かす（招待 + チーム稼働管理）

**目的**: チームメンバーを招待し、メンバーの稼働入力 → 管理者の請求書発行フローを完結させる。

**ゲート条件（着手前提）**: Phase 1 のゲート条件を満たしていること + SendGrid メール基盤が稼働していること。

| 項目                                                       | 対象ペルソナ | 概要                                   |
| ---------------------------------------------------------- | ------------ | -------------------------------------- |
| SendGrid メール基盤                                        | B, C         | 招待メール送信の基盤                   |
| 招待機能（better-auth invitation API）                     | B, C         | メール招待 → 承認で参加                |
| 招待承認ルート                                             | C            | `/invite/:invitationId`                |
| Settings/Members の改修                                    | B            | 直接追加 → 招待ベースに移行            |
| セットアップ完了 + TODO ガイダンス                         | B            | Step 3: 完了画面 + ダッシュボード TODO |
| ダッシュボード TODO バナー                                 | B, C         | 管理者向け / スタッフ向け              |
| 招待 UI（Settings/Members + ダッシュボード TODO から誘導） | B            | メンバー招待は初回セットアップ後に案内 |

**成功指標（KPI）**:

| 指標                 | 目標値  | 計測方法                                             |
| -------------------- | ------- | ---------------------------------------------------- |
| 招待→参加完了率      | ≥ 80%   | 招待承認数 / 招待送信数                              |
| 参加→初回稼働入力率  | ≥ 70%   | 参加後 7 日以内に稼働を 1 件以上入力したメンバー比率 |
| メンバー月次入力率   | ≥ 90%   | 月末時点で当月の稼働を 1 日以上入力したメンバー比率  |
| 管理者の月次処理時間 | ≤ 30 分 | 全メンバーの稼働確認 〜 請求書一括生成の所要時間     |

**次 Phase へのゲート条件**: 2 名以上のメンバーが 1ヶ月間 Phase 2 の機能で稼働入力を完了できること。

### Phase 3: D を構築する（クライアントポータル）

**目的**: クライアント企業が委託先の稼働と請求書を一箇所で確認・承認できる。

**ゲート条件（着手前提）**: Phase 1〜2 のゲート条件を満たしていること。具体的には:

- 請求書ステータス管理が運用されている（下書き / 発行済み / 入金済み）
- メンバーの稼働確定フローが動作している
- SendGrid メール基盤が稼働している

| 項目                        | 対象ペルソナ | 概要                     |
| --------------------------- | ------------ | ------------------------ |
| 稼働確定フロー              | B, C         | スタッフの月次稼働確定   |
| クライアント招待 + ポータル | D            | 稼働閲覧 + 請求書受領    |
| 請求書承認フロー            | D            | 承認ステータスの追加     |
| 通知・レポート              | D            | メール通知、月次レポート |

**成功指標（KPI）**:

| 指標                       | 目標値 | 計測方法                                       |
| -------------------------- | ------ | ---------------------------------------------- |
| ポータル招待→アクセス率    | ≥ 70%  | 招待後 7 日以内にログインしたクライアント比率  |
| 請求書閲覧率               | ≥ 90%  | 発行済み請求書のうちクライアントが閲覧した比率 |
| 「請求書まだ？」催促の削減 | ゼロ   | クライアントからの催促連絡の有無（定性）       |

## CLI 同期の品質指標

CLI 同期は A の体験の中核であり、以下の指標で継続的に品質を監視する。

| 指標                 | 定義                                                           | 目標値 | 計測頻度 |
| -------------------- | -------------------------------------------------------------- | ------ | -------- |
| 同期成功率           | invoiceai sync がエラーなく完了した回数 / 総実行回数           | ≥ 95%  | 毎月     |
| 手修正率             | sync 後に Web UI で稼働時間を手動修正した日数 / sync 対象日数  | ≤ 20%  | 毎月     |
| 月次処理時間         | 月末の sync 実行 〜 請求書 PDF 生成完了までの実測時間          | ≤ 15分 | 毎月     |
| アクティビティ網羅率 | sync で収集されたアクティビティ数 / 実際の作業日数             | ≥ 90%  | 毎月     |
| セットアップ時間     | 新規リポジトリで初回 invoiceai 〜 初回 sync 完了までの所要時間 | ≤ 3分  | 初回     |

これらの指標は Phase 1 完了後、毎月の請求フロー時に自分で計測・記録する。目標値を 3ヶ月連続で達成した時点で A の DoD 達成とする。
