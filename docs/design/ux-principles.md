# UX 設計原則

InvoiceAI 全体に適用する UX の原則。

## 1. オブジェクト指向 UI（OOUI）

**ユーザーはまず対象（名詞）を選び、次に操作（動詞）を選ぶ。**

- 画面の主役はオブジェクトの一覧（クライアント、経費、稼働時間、請求書）
- 操作はオブジェクトから（クリック、DropdownMenu、インライン編集）
- フォームは常設ではなく、操作に応じて出現する

適用例:

- 経費設定: ExpenseCard が主役。DropdownMenu から編集・削除
- クライアント一覧: クライアント行をクリック → 詳細
- 稼働時間: 日付行をクリック → 入力ポップオーバー

**アンチパターン:** フォームが画面の大半を占め、データ一覧が脇役になっている

## 2. プログレッシブ・ディスクロージャー

**必要になるまで見せない。**

- 初期状態は最小限。操作に応じて詳細が出る
- 追加フォームはボタンを押したときだけ表示（ダイアログ or インライン展開）
- 高度な設定は折りたたみや「詳細設定」リンクの後ろに
- 入力フィールドが5つ以上あるなら、ステップ分割を検討

適用例:

- 経費の内訳: Collapsible で展開。閉じた状態がデフォルト
- GCP Billing 設定: 従量課金追加フローの中にステップとして組み込み
- 為替レート: 自動値を表示。「変更」を押したときだけ入力欄

## 3. 視覚的階層

**最も重要な情報が最も目立つ。**

- 金額: `text-lg font-semibold tabular-nums` — 最も目立つ
- タイトル/名前: `font-medium` — 中程度
- 補足情報（通貨、税率、日付）: `text-xs text-muted-foreground` — 控えめ
- アクションボタン: hover 時のみ表示（`opacity-0 group-hover:opacity-100`）
- 主ボタンは1画面に1つ。副操作はゴーストボタンかテキストリンク

## 4. 空状態（Empty State）

**データがないときこそ丁寧に。**

- 何を・なぜ・どうやって追加するかを説明する
- 「追加」ボタンを目立たせる
- できれば具体例を見せる（「例: サーバ通信費 月$45」）

適用例:

- 経費未設定: 「経費はまだ登録されていません」+ ガイダンス + 2つの入口ボタン
- クライアント未登録: 設定ページへの誘導

## 5. フィードバック

**操作の結果が即座にわかる。**

- 保存 → 一覧に即反映（useFetcher の revalidation）
- 請求書作成 → 結果（番号、金額）を同画面で表示
- エラーはフィールドの隣に表示
- 破壊的操作（削除）は confirm ダイアログ

## 6. 導線の明確さ

**今どこにいて、どこに行けるか。**

- パンくずリスト or 「← 戻る」リンク
- 関連画面への相互リンク（請求書作成 ⇄ 経費設定）
- タブで現在位置を示す
- 設定変更後に元の画面に戻れる
